# Spec: Local Codex Sessions Browser

## Objective

为本地单人自用的 Euphony 增加一个 Codex sessions 浏览器，让使用者打开本地 Euphony 后可以自动读取本机 Codex CLI sessions，并按“项目 -> session 标题 -> session 内容”的维度浏览，不再需要手动上传 rollout JSONL 文件，同时可以单条或批量永久删除后端扫描到的 rollout JSONL 源文件。

目标用户是本机使用 Euphony 查看自己 Codex 历史记录的开发者。第一版以真实本机 `CODEX_HOME` 数据可用为成功标准，不把该能力设计成多用户、远程托管或上游通用部署功能。

### Core User Stories

- 作为本地用户，我打开 Euphony 后默认看到 Codex sessions 浏览器，而不是 demo 数据。
- 作为本地用户，我可以按项目查看 sessions；项目由 session `cwd` 推导出的 Git 根目录决定，找不到 Git 根目录时回退到原始 `cwd`。
- 作为本地用户，我可以在某个项目下看到 session 标题、预览、路径、更新时间和归档状态。
- 作为本地用户，我点击一个 session 后，可以看到完整 Codex session 内容，渲染质量至少不低于当前手动上传 JSONL 的体验。
- 作为本地用户，我可以从 session 详情区单条永久删除该 session 对应的 rollout JSONL 文件。
- 作为本地用户，我可以在当前项目的 session 列表里勾选多个 session 并批量永久删除对应的 rollout JSONL 文件。
- 作为本地用户，我可以手动刷新，让新产生的 sessions 出现在浏览器中。
- 作为本地用户，我仍然可以通过现有入口加载 URL、剪贴板或本地文件，避免破坏原有工作流。

### Success Criteria

- 启动本地 FastAPI 后端和 Vite 前端后，访问 `http://localhost:3000/` 默认进入本地 Codex sessions 浏览器。
- 后端能扫描 active sessions 和 archived sessions，并返回至少包含项目列表、session 摘要、单个 session 完整内容，以及永久删除入口的 API。
- 使用真实本机 `~/.codex` 或 `CODEX_HOME` 数据验证：可以列出项目、列出项目内 sessions、打开至少一个 session 并渲染内容。
- 归档 sessions 会出现在列表中，并带有明确的 archived 状态。
- 前端不能让用户传任意本地路径给后端读取；session 内容只能来自后端扫描出的 Codex rollout 白名单。
- `pnpm run build` 通过。
- 后端解析和 API 行为有 Python 测试覆盖，至少覆盖 active rollout、archived rollout、缺失 `cwd`、无效 JSONL 行和路径越界拒绝。

## Tech Stack

- Frontend: TypeScript, Vite, Lit Web Components, Shoelace, existing Euphony components.
- Backend: Python, FastAPI, Pydantic, standard library `pathlib` / `json` / `os` / `sqlite3` only if later needed.
- Package manager: `pnpm@10.25.0` as declared in `package.json`.
- Testing: existing Python tests under `tests/`; frontend build/type checks through `pnpm run build`.

Dependency policy for this feature is “尽量零新增”:

- Backend should parse rollout JSONL and `session_index.jsonl` with Python standard library first.
- Frontend should reuse existing Lit/Shoelace patterns and existing Codex renderer.
- Any new dependency must be justified before adding, including why existing code or standard library is insufficient.

## Code Style

Follow the existing repository style:

- TypeScript uses single quotes, semicolons, strict typing, Lit decorators, and explicit state fields.
- Python uses type annotations, Pydantic models for API responses, clear FastAPI errors, and small pure helpers for parsing.
- Comments should be written in Simplified Chinese when comments are necessary. Prefer clear names and small functions over comments that explain obvious code.
- Error messages must include context: what operation failed, which input was involved when safe to show, and what the user can do next.
- Keep implementation conservative; do not add abstraction until the third repeated pattern.

Example backend style:

```python
from pathlib import Path
from pydantic import BaseModel


class CodexSessionSummary(BaseModel):
    id: str
    title: str
    preview: str
    cwd: str | None
    project_id: str
    rollout_path: str
    archived: bool


def resolve_codex_home() -> Path:
    codex_home = Path(os.environ.get("CODEX_HOME", Path.home() / ".codex"))
    return codex_home.expanduser().resolve()
```

Example frontend style:

```typescript
interface CodexSessionSummary {
```

## Testing Strategy

Test behavior, not implementation details.

### Backend Tests

Add Python tests for the Codex session scanner and API behavior:

- Parses active rollout files under `sessions/YYYY/MM/DD/`.
- Parses archived rollout files under `archived_sessions/`.
- Extracts `id`, `cwd`, timestamps, title fallback, preview fallback and archived status.
- Groups sessions by Git root when fixture directories contain `.git`, and falls back to exact `cwd` when no Git root exists.
- Handles missing `session_meta.cwd` by assigning `Unknown project`.
- Rejects session reads when the requested id is unknown or not in the scanned whitelist.
- Handles malformed JSONL lines with clear errors or controlled skipping, as defined during implementation.

### Frontend Verification

Use `pnpm run build` as the required TypeScript and production build check.

Manual browser verification is required for this feature because it changes the primary user flow:

- Start backend on `127.0.0.1:18020`.
- Start Vite on `localhost:3000`.
- Open the app and confirm it defaults to the local Codex sessions browser.
- Confirm project list, session list, session detail rendering and manual refresh.
- Confirm URL/file/clipboard loading remains accessible.

### Coverage Expectations

- Parser and path safety behavior should be covered by automated tests.
- UI state transitions can be validated manually for MVP unless a test framework is already present.
- No known warnings from build, type checks or tests should remain. If a warning cannot be eliminated, add a narrow inline ignore with justification.

## Boundaries

### Always Do

- Read Codex sessions through the local backend only; frontend must not directly access the filesystem.
- Treat rollout JSONL as the source of truth for session content.
- Include both active and archived sessions in the local browser.
- Group projects by Git root derived from session `cwd`, with fallback to exact `cwd`.
- Use manual refresh for MVP; avoid polling or file watching.
- Keep list/detail APIs read-only, and expose a dedicated delete endpoint that permanently removes rollout JSONL files from the backend whitelist.
- Preserve existing URL, clipboard and local file loading workflows.
- Run `uv run --with pytest pytest` and `pnpm run build` before committing implementation changes.
- Keep generated `lib/` untouched.

### Ask First

- Adding any new dependency.
- Reading SQLite `state_5.sqlite` as a primary metadata source.
- Adding write operations such as rename, archive or unarchive. The session delete flow described above is in scope.
- Changing the public web component API.
- Changing CI, package manager configuration, build scripts or deployment scripts.
- Changing the physical layout of `CODEX_HOME`.
- Adding background polling, file watchers or persistent caches.

### Never Do

- Never modify, move or rewrite Codex rollout files outside the explicit delete flow described above, and never rewrite `session_index.jsonl`.
- Never allow frontend-supplied arbitrary local paths to be read by the backend.
- Never expose this local file-reading API as a remote multi-user feature without a new security design.
- Never store secrets, API keys or full session contents in browser localStorage.
- Never silently swallow parse or filesystem errors.
- Never document features that are not implemented.
- Never edit files under `lib/`.

## Open Questions

- Should malformed JSONL lines fail the whole session read, or should the API skip bad lines and surface a warning?
- Should default session ordering be strictly `updatedAt desc`, or should active sessions sort ahead of archived sessions?
- Should the first implementation include a small in-memory cache for scan results, or should every manual refresh rescan from disk?

Implementation must not begin until this spec is reviewed and confirmed.
